================================================================================
                      KEY CHANGES SUMMARY
================================================================================

### Import Changes
```javascript
// OLD: Fake base44 SDK
import { base44 } from '@/api/base44Client';

// NEW: Real backend APIs
import { entities, apiPost } from '@/api/entities';
```

### API Pattern Changes

Pattern 1: List with sorting
  OLD: base44.entities.Entity.list('-created_date', 50)
  NEW: entities.Entity.list('-created_at', 50)

Pattern 2: Create
  OLD: base44.entities.Entity.create(data)
  NEW: entities.Entity.create(data)

Pattern 3: Update
  OLD: base44.entities.Entity.update(id, data)
  NEW: entities.Entity.update(id || _id, data)  // Handle both ID types

Pattern 4: LLM/Agent calls
  OLD: base44.integrations.Core.InvokeLLM({ prompt, response_json_schema })
  NEW (Chat):    apiPost('/api/agents/chat', { message, business_id, session_id })
  NEW (Agent):   apiPost('/api/agents/run', { task, business_id, skip_confirmation })
  NEW (Scan):    apiPost('/api/intelligence/scan', { business_id, focus })
  NEW (Voice):   WebSocket ws://localhost:3001/ws/voice

### Response Shape Changes

Chat response:
  ✅ { reply, session_id, messages }

Agent response:
  ✅ { agentTaskId, intent, steps, final_summary, records_created }

Scan response:
  ✅ { agentRunId, opportunities, summary, data_freshness }

Voice message:
  ✅ { type: 'partial'|'final'|'error', text, confidence }

### MongoDB ID Handling

Pattern:
  ```javascript
  // Always use: id || _id
  entities.Lead.update(lead.id || lead._id, { status })
  ```

Reason: MongoDB uses _id internally, but some responses may map to id
